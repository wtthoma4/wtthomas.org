Workshop Companion Instructions

This article contains step-by-step instructions, with screenshots, for creating a GitHub Pages personal portfolio site. Exact instructions for the initial set up and configuration of three popular themes, Al Folio, Chirpy, and Hydejack, are included.

Thanks to NC State’s College of Engineering, NC State’s Association for Computing Machinery chapter, Tiva Rocco, and Dr. Sarah Heckman for helping make this workshop happen!

Selecting a Theme

For the three themes we are supporting for this workshop, consider the following recommendations:

For a portfolio or project focused site, choose Al Folio or Hydejack.

For a blog or chronological site, choose Chirpy or Hydejack.

Chirpy is the simplest and easiest to configure, followed by Hydejack, followed by Al Folio. You can view samples of each theme using the following links:

https://alshedivat.github.io/al-folio/

https://chirpy.cotes.page/

https://hydejack.com/docs/

Al Folio

First, navigate to https://github.com/alshedivat/al-folio. In the top-right corner of the repository, click Use this template > Create a new repository.

Name the repository username.github.io. Make sure it is Public and click Create repository.

For Al Folio to work, we must grant permissions to GitHub Actions. In your repository, go to Settings > Actions > General. Toward the bottom, update the Workflow permissions and click Save.

Next, navigate to _config.yml in your repository and click edit. Update the url and baseurl attributes as shown below. Click Commit changes... in the top-right corner and save your changes.

After your website is built (the yellow ball turns into a green check), confirm that a new branch has been created in the repository called gh-pages.

We need to tell GitHub Pages to use this new branch. In your repository, click Settings and go to Pages in the left-hand sidebar. Ensure Source is set to Deploy from a branch. Under the branch selector, change it from None to gh-pages and click Save.

Next, let’s confirm we can edit and rebuild the site with our changes. Navigate to _config.yml in your repository and click edit. Update the title and name attributes as shown below. Click Commit changes... in the top-right corner and save your changes.

After the website rebuilds, refresh and confirm your changes took effect.

Success! Now, the real works begins. Continue making changes to the configuration according to your goals for the site. The default pages contain instructions for making changes, plus samples of how to create blog posts, projects, and more.

Chirpy

Navigate to https://github.com/cotes2020/chirpy-starter. In the top-right corner of the repository, click Use this template > Create a new repository.

Name the repository username.github.io. Make sure it is Public and click Create repository.

Now that we have a repository, we need to enable GitHub Pages. In your repository, click Settings and go to Pages in the left-hand sidebar. Ensure Source is set to Deploy from a branch. Under the branch selector, change it from None to main and click Save.

Navigate to _config.yml in your repository and click edit. Change the url to username.github.io. Click Commit changes... in the top-right corner and save your changes.

After your website is built (the yellow ball turns into a green check), navigate to username.github.io. You should see the default Chirpy website.

Now, we can start configuring our website. First, let’s change the name and tagline. Navigate to _config.yml and click edit. Change the title and tagline attributes to whatever you want! Click Commit changes... in the top-right corner and save your changes.

Once the website is rebuilt, reload the site and you see your changes.

Your website is running! Now, the real work begins. Use _config.yml to adjust the other aspects of the website as you see fit. Write new articles in .md (markdown) format and upload them to the _posts folder.

For long-term work, I recommend following the Chirpy developers’ instructions to set up a development environment on your local machine, found here https://chirpy.cotes.page/posts/getting-started/. The developers’ have also written instructions to create a new article, found here https://chirpy.cotes.page/posts/write-a-new-post/.

Hydejack

Navigate to https://github.com/hydecorp/hydejack-starter-kit. In the top-right corner of the repository, click Use this template > Create a new repository.

Name the repository username.github.io. Make sure it is Public and click Create repository.

Now that we have a repository, we need to enable GitHub Pages. In your repository, click Settings and go to Pages in the left-hand sidebar. Ensure Source is set to Deploy from a branch. Under the branch selector, change it from None to master and click Save.

Navigate to _config.yml in your repository and click edit. Comment out (add a # before) the theme: attribute. Un-comment (remove the # before) the remote_theme: attribute. This will allow GitHub to pull the theme properly. Click Commit changes... in the top-right corner and save your changes.

After your website is built (the yellow ball turns into a green check), navigate to username.github.io. You should see the default Hydejack website.

Now, we can start configuring our website. First, let’s change the name and tagline. Navigate to _config.yml and click edit. Change the title and tagline attributes to whatever you want! Click Commit changes... in the top-right corner and save your changes.

Once the website rebuilds, refresh the website and you shold see your personalized version.

Success! Now, the real works begins. Continue making changes to the configuration according to your goals for the site. The default pages contain instructions for making changes, plus samples of how to create blog posts, projects, and more.

GitHub Codespaces

Instead of configuring a local machine to work on your website, we can use GitHub Codespaces to do our work in the cloud! This should be done after you have successfully built a working version of the website.

To open a codespace, click Code in your repository and choose the Codespaces tab. Then, click to open a codespace on main.

Here, I’m using the Al Folio example page. After loading into my codespace, you may notice it looks a lot like VSCode. We’re basically running VSCode on GitHub’s servers and accessing it via a web browser. After opening the codespace, you may notice some terminal activity as Jekyll automatically starts and builds this temporary version of our website. Once the build is finished, you will see a prompt to open the website in the bottom right corner.

If you open this side-by-side with your codespace, you can make changes and watch them happen on your site in real time. This is much faster than making changes in the repository and waiting for the site to be rebuilt in GitHub Pages every time!

For example, I can open the _config.yml file in my editor and change the name to John Smith. Jekyll will automatically rebuild the whole site after I save my changes, and after the rebuild is finished, the website will reload with my changes:

Once you feel ready to deploy your changes on your public site, you can use the codespace source control to commit directly to your repository.

Once the changes are built in your public site, refresh and check out your changes!
