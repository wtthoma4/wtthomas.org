Does TEE Solve AI Privacy?

Maple offers SOTA open-source AI models with a zero-knowledge guarantee. Is it legit?

Posted May 31, 2026
By Taylor Thomas
29 min read
Does TEE Solve AI Privacy?
Does TEE Solve AI Privacy?

What if I want to ask AI a legal question, a financial question, or a medical question? In practice, many people say things to AI chatbots they would never dare utter aloud, and maybe never have. This is about as close to giving Big Tech direct access to your brain as we’ve ever come. Even if one were to trust a given Big Tech company, would you feel comfortable publishing your chat history for anyone to read? At the time of writing, an AI platform has yet to experience a data breach (that we know of), but this is inevitable. Do you want a catalog of your thoughts read aloud in town square?

The solution is to run AI models on your own hardware. The obstacle used to be that open-source models were not comparable to frontier Big Tech models, but this gap is closing rapidly. Kimi K2.6 is hot on the heels of Big Tech, and Kimi 3.0 is right around the corner. The obstacle is now solely cost. Running an AI server requires storage, memory, and compute that far exceeds the utility a typical person will derive from their AI, especially without 24/7 use.

MapleAI (now Maple Research) promises a private way to interact with the best open-source AI models. Maple claims that only users can view their chat history, and that users can independently verify a server is running open-source code. The value proposition is simple and compelling. I give Maple money, Maple gives me efficient cloud compute for excellent AI models without being to spy on me.

But is that true? Is it even possible?

Let’s investigate.

Chapter 1: Architecture

Maple is open-source, so let’s start by looking at what components Maple uses and how they interact.

The Maple codebase is the client-side application(s). This includes the web app, the iOS and android apps, and the various desktop apps. Opensecret is the server your client communicates with. Opensecret runs in the cloud, specfically on AWS Nitro EC2 instances. Those cloud containers are special, because they offer hardware based attestation, which is very important to this use case. Maple uses the Opensecret SDK to handle all of the communication between the Maple client and the Opensecret server.

This is great, but neither Maple nor Opensecret have any AI in them… where’s my LLM? The problem is the Opensecret server runs on Nitro enclaves, which don’t have GPUs. The solution is a secondary TEE, running on a GPU, which can performantly run an AI inference engine like vLLM. At the time of writing, Opensecret implements two such providers: privatemode.ai and tinfoil.sh.

This begs an interesting question, why use Opensecret at all? Why not access my hardware-attested secure inference engine running on a GPU TEE directly? The purpose of the Opensecret middleman comes from the limitations of hardware-attested cloud servers. CPU and GPU enclaves cannot perform the same work. For authentication, billing, session management, and database interactions, I need a CPU. For AI inference, I need a GPU. In order to get both, I must daisy-chain TEEs together into one system. In practice, my Maple client cryptographically verifies that an Opensecret server is running the open-source code I’ve inspected, while the Opensecret server verifies that my AI inference engine is running the open-source code I’ve inspected.

Chapter 2: Sessions

A session is a short-lived cryptographic relationship between a client and server. To understand Maple’s security guarantees, we have to start with how sessions are handled between the Maple client and the Opensecret enclave. Every session starts the same way:

  • The client cryptographically proves it’s talking to a real enclave.
  • The two sides negotiate a shared symmetric key.
  • Every request/response body that could carry sensitive content is encrypted with this key before transmission.

Login Flow

When you launch Maple and log in, four things happen in order: attest, handshake, authenticate, store tokens. What makes this different from a normal HTTPS login is the first two steps. The client uses hardware attestation to prove that it’s not just talking to a server, but to a specific, known piece of code running inside an AWS-sealed box.

First, the client demands a proof of identity from the server. The client generates a random UUID (128 bits, of which 122 are random) to use as a nonce that proves freshness.

The client has a cryptographic proof, rooted in AWS hardware, that it is talking to a specific known build of Opensecret inside a real Nitro enclave. Everything that follows can now safely rely on the X25519 public key included in the attestation document.

From this point onward, the remaining handshake is what we’d expect from a modern web application. The client encrypts the user’s password under the session key; the server decrypts it, verifies it against an Argon2 hash that is itself stored encrypted under the enclave key, then issues two JWTs: an access token and a refresh token. Every request body that could carry sensitive content (prompts, replies, credentials) is encrypted under the session key derived from the attested X25519 keypair, so it remains readable only by the enclave. This can be reviewed in OpenSecret-SDK/src/lib/encryption.ts, OpenSecret-SDK/src/lib/encryptedApi.ts, opensecret/src/web/encryption_middleware.rs, opensecret/src/web/login_routes.rs, opensecret/src/main.rs, and opensecret/src/models/users.rs.

Inference Flow

At this stage, we know that the client and the enclave have an end-to-end-encrypted session and communicate securely. I’ve logged in and now I want to prompt an AI model, how does that happen?

When you send a prompt, Opensecret decrypts your prompt inside its enclave and somehow must get that plaintext to a GPU. A Nitro enclave has no GPU, so the prompt has to leave Opensecret for a second confidential machine, which we will call TEE2. Opensecret now does what our client did at login: it uses hardware attestation to prove TEE2 is a specific, known build of an inference server running on genuine confidential hardware.

If I can just modify some environment variables and the code supports sending prompts to Big Tech, how do I know where my plaintext prompt is going? Furthermore, could it be connecting to any inference engine and still pass the source code hardware attestation?

My best guess is that this route exists for development and testing reasons. However, we don’t have to trust that the server operator is using the correct environment variables. The enclave startup script, entrypoint.sh, controls these environment variables. This entrypoint file is part of the enclave image, and therefore is included in the PCR0 that our client uses at login to verify the code running on the Opensecret server. Both proxies run as separate processes within the Opensecret enclave. Opensecret talks to each proxy over plain HTTP on loopback; that is safe only because both processes live inside the same attested enclave. The first part of this check is to establish what should be running. Sigstore is a public software-signing system. This is an alternative to the hard-coded allowlist that Maple<->Opensecret uses. Sigstore is automated while still being transparent and tamper-resistant. The Tinfoil proxy fetches the allowlist using Sigstore and GitHub.

It is worth pausing on what the hardware attestation here actually is, because it is not the same mechanism as the AWS NSM attestation from the login flow, even though it plays the same role. The shape is shared: a hardware root of trust signs a document that says “here is a measurement of the code that booted, and here is some caller-chosen data I will bind to it,” and you verify by chaining the signature to a manufacturer root and checking the measurement against an expected value. What differs is the manufacturer, the measurement model, and how the caller-chosen data works.

Tinfoil first fetches the document and learns the platform from it. The GPU box exposes an endpoint that returns a small JSON object whose Format field declares which silicon produced it.

The self-declaration is not a trust hole, because the format only chooses which vendor PKI the signature is forced to satisfy. On the AMD path, the SEV-SNP report must verify against AMD’s chain. On the Intel path, the TDX quote must verify against Intel’s chain. A box that lies about its platform simply has to forge the other vendor’s signature instead, which it cannot do; and either way Tinfoil still cross-checks the measurement against the Sigstore-pinned expected value.

The caller-chosen data is where this most differs from NSM. The login attestation passes an explicit nonce and public_key into the NSM request, so each session gets a fresh challenge. SEV-SNP and TDX instead expose a single fixed 64-byte report_data field that the guest fills once when it generates the report. The Tinfoil inference enclave puts its TLS public key there, and the verifier reads it back out of the hardware-signed report.

This is the mechanical reason the inference path is “attest once, pin the key” rather than “fresh nonce per request” like login: the hardware report binds a long-lived TLS identity, not a per-call challenge, so the strong guarantee is delivered by pinning every subsequent connection to that attested key rather than by re-attesting each prompt.

The final destination of our prompt is vLLM. At boot, the Tinfoil server verifies which version of vLLM is included in the server image. vLLM is specified in a config file, and that config’s SHA-256 is carried on the kernel command line, which is part of the same measured boot covered by AMD or Intel hardware attestation. To change the engine, you must change the config and change the config’s hash, which changes the measured hardware attested value. On top of that, model weights get a second dm-verity integrity check.

If vLLM sees our plaintext request (which it must), then our trust chain can be broken at the last step if vLLM is logging it or sending it off to a third-party. vLLM’s logging and usage-statistics behavior is controlled by its launch arguments, and those arguments live in tinfoil-config.yml, whose SHA-256 is carried on the measured kernel command line, so the exact vLLM invocation is measured by hardware attestation. The other structural guarantee is that the confidential VM boots a read-only, integrity-checked root filesystem: the kernel command line carries readonly=on and a dm-verity roothash=, both folded into the same SEV-SNP/TDX launch measurement, so vLLM cannot persist a prompt to disk or silently run modified code.

Storing Chats

By default, Maple stores your conversation history server-side so you can return to it from any device. This sounds like it should torpedo the privacy story, but the schema is built around the same heuristic as the rest of the system: every column that contains user content is stored as ciphertext, and the only key that can decrypt it lives inside the enclave.

The database schema for chat content uses column names suffixed _enc:

  • prompt_enc — the prompts you sent
  • content_enc — message bodies
  • name_enc — conversation titles
  • metadata_enc — anything else attached to a response
  • attachment_text_enc — any file/image text you’ve uploaded

When a row is read, the enclave runs decrypt_content(&user_key, msg.content_enc.as_ref()), where user_key is a per-user secret derived inside the enclave from material that never leaves the enclave. If anyone outside the enclave were to dump the database, they would see a row of opaque bytes and nothing else. Even the row count and approximate timing leak only metadata, not content.

pub fn decrypt_content<T>(
    key: &SecretKey,
    encrypted: Option<&Vec<u8>>,
) -> Result<Option<T>, EncryptError>
where T: DeserializeOwned
{
    let Some(encrypted) = encrypted else { return Ok(None); };
    let decrypted_bytes = decrypt_with_key(key, encrypted)?;
    let value = serde_json::from_slice(&decrypted_bytes)
        .map_err(|e| EncryptError::DeserializationFailed(e.to_string()))?;
    Ok(Some(value))
}

This is the same trust chain we’ve been tracing extended one more hop. The database is not in the enclave, so the database doesn’t get to see plaintext. The enclave acts as the gatekeeper, decrypting on read and encrypting on write, and the keys live in measured-and-attested memory.

Chapter 3: Proofs

We can inspect the code for all parts of this system. When we generate a plaintext prompt, we can perfectly trace its visibility and flow through the architecture. At every point, the next destination of a prompt or reply is deterministic. This observation offers strong security and privacy guarantees, but they rest on the ability of end users to compare the hashes the source code checks with the hashes of the codebase when they build it themselves.

The system does emit one outbound stream: HMAC-signed billing events containing token counts, model name, and a request ID, never prompts or replies, to a destination fixed in the attested tinfoil-config.yml and constrained by an attested nftables egress allowlist.

Reproducible Builds

In order to confirm that the system is running the code in the codebases we inspected, we need to find every point where a hash is checked and compare the “good” value with a value we arrived at ourselves. If I build the code on my laptop and the hashes don’t match the ones the system is using, I have no idea what code is actually running.

So let’s enumerate every hash the system checks, and ask, for each one: can I, on my own machine, arrive at the same value?

There are really only two numbers that matter, because everything else folds into them. The Maple client checks one measurement: the PCR0 of the Opensecret enclave. The Opensecret enclave checks one measurement: the SEV-SNP (or TDX) launch measurement of the Tinfoil model box. Every other hash we’ve seen is an input to one of those two:

If I can independently reproduce the two measurements at the bottom of the “folds into” funnel, every input above them is verified for free, because changing any input changes the measurement.

The Scripts

Doing a one-time reproduction is a significant and valuable result, but knowing that someone is honest right now proves nothing about whether they will be honest tomorrow. Therefore, I wanted to automate all the work I do for reproduction, and make it as hands-off as possible. My goal is that any auditor or user should be able to download a script, run it, and check for themselves whether Maple is being honest. The script should do the absolute maximum possible configuration for you, only requiring user intervention when unavoidable. Friction is bad.

It only took me a few hours to get a working version of a script that successfully checked and installed dependencies, cloned all of the source material, configured Nix, fetched the attestation and AWS certificates, and built the enclave images from source. However, the PCR0s did not match. That’s the whole point! No need to break out the pitchforks though, if your first attempt works you’re more likely to have done something wrong than done something right.

To ensure that my build procedure was correct, I began with a commit dated a while ago. This allows me to check my process, and protects me from a push-before-we-update-the-attestation situtation in the commit history. For a while, I was getting PCR0s that didn’t exist anywhere in the Openscret repository.

What really killed me was build speed. The Opensecret enclave image is built for ARM Linux because production runs on AWS Graviton (aarch64) Nitro instances. I only have x86 hardware, so the build has to happen under emulation to produce the same bytes the live system is running. Opensecret is built with Nix, which has built-in support for QEMU emulation, but emulated builds are slow. Every tweak or attempt to correct any build issues was therefore followed by several hours of emulated Nix builds, even with caching.

After much tweaking and trial-and-error, I finally dialed in my OS and tools, and produced a PCR0 of d4218d625c17a7bbff6dd95c8ccea98d851284744c43cfbae98e86a7d7d504c1dc80908a6d34eab6f859262502db3366 from commit 416c03e. This is a deprecated PCR0, but it was used as a production attestation in the past. This proved that my approach and configuration were correct, and also proved that Maple was using the code they claim to use, and that Opensecret is byte-reproducible from source.

But I’m a stubborn fella, and I want the live PCR0. I don’t care what Maple was running yesterday, I care what is running right now. Maple does not use releases for Opensecret, so the closest thing we have to documentation about what PCR0 corresponds to what commit is the commits that change the PCR0 values themselves. We know that the live production PCR0 is listed in pcrprod.json, and we can see when this value changes via the commit history on that file. Today is 2026-05-31. The current live PCR0 is 3c374a4b3c4f4cd5c5a6cc28d89cc2462fdd129eac5cd0eacf7fc78b98af11e9a08750a76cd1c1ab75c2715300530a26. This value was introduced at commit b2eee16 on 2026-05-06 and the previous PCR0 of d4218d625c17a7bbff6dd95c8ccea98d851284744c43cfbae98e86a7d7d504c1dc80908a6d34eab6f859262502db3366 was deprecated. By checking out to b2eee16, the same script and environment that produced the previous matching PCR0 produced 3c374a4b3c4f4cd5c5a6cc28d89cc2462fdd129eac5cd0eacf7fc78b98af11e9a08750a76cd1c1ab75c2715300530a26, which is the live production attestation! Victory!

However, is this victory certain in the future? I succeeded because the commit that updated pcrProd.json built to the live PCR0, but this may not always be the case. Opensecret does not use releases. Therefore, there is no clear delineation of what to build. The Opensecret codebase is updated regularly, but not every GitHub commit affects the PCR0 measurement. Roughly, a commit changes PCR0 only if it changes the inputs to the EIF derivation: the Rust source under src/, Cargo.lock, rust-toolchain.toml, flake.nix (when the kernel config or build pipeline change), entrypoint.sh, or the committed binaries under nitro-bins/, continuum-proxy, and tinfoil-proxy/dist/. Commits that touch only prcProd.json, pcrProdHistory.json, documentation, CI workflows, tests, or the README do not change the PCR0, even though they advance the master branch. This is why two adjacent commits can have identical PCR0s and a long stretch of main can share a single deployed measurement. Therefore, I included a walk-back mechanism in my script which looks for the live attestation PCR0 and, if the first candiate commit fails, works around it to try and find a commit that builds to the expected PCR0.

The tools I used are available at wtthoma4/maple-tee-audit. I will do my best to keep them fresh.

Together these prove both halves of the claim: the system is reproducible from public source today, and it was reproducible from public source for at least one prior deploy. The live attestation document also passed the full cryptographic chain; its certificate validates to the AWS Nitro root, its COSE_Sign1 signature is valid, and the nonce I generated in the run was echoed back inside the signed payload, so I know the attestation is fresh and not replayed.

There is one documented gap, in nitro-bins/: two AWS-derived prebuilt binaries (kmstool_enclave_cli, libnsm.so) ship as committed bytes in the Opensecret repo. AWS’s published recipe for rebuilding them has a transitive dependency that now requires a newer Rust edition than the recipe pins, so the recipe doesn’t build today. This doesn’t weaken the trust chain. Those bytes are committed and folded into PCR0, so my reproduced PCR0 covers them exactly it just means I can’t independently rebuild them. Notably, OpenSecret has already started addressing this: PR #191 (titled “Restore reproducible nitro-bins build”) is merged into main at the time of writing.

The notable limitation on the Tinfoil side is that I have not yet rebuilt the cvmimage itself from source. Sigstore + Rekor is meaningful trust, it ties each published artifact to a short-lived signing certificate bound to the identity of the GitHub-hosted runner attached to the tinfoilsh/cvmimage repository, and the signature is recorded in an append-only public log. This is a notch weaker than building from source. I attempted the rebuild and ran into a few recipe-portability issues on Tinfoil’s side (notably, undeclared build-tool dependencies and package-version conflicts when installing from the pinned Ubuntu snapshot); none of them are security gaps in the running system.
