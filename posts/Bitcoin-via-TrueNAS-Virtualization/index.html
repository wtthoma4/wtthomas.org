Bitcoin via TrueNAS Virtualization

Don't trust, always verify.

Posted Mar 17, 2026 Updated Mar 19, 2026
By Taylor Thomas
30 min read
Bitcoin via TrueNAS Virtualization
Bitcoin via TrueNAS Virtualization

I need a Bitcoin node. I have a TrueNAS server. Let’s figure it out!

Motivation

I am working on a project that requires extensive interactions with the Bitcoin mainnet, testnet, and a mempool. I need a box in my development environment and a standardized build procedure. I decided to document the whole process!

Available Hardware

I have a TrueNAS server with plenty of CPU and RAM headroom to take on this project. The sketchy area is disk space. I bought a 1TB SSD for this project, and that will suffice for now, but within a few years a 1TB drive will no longer hold the unpruned blockchain plus the chainstate data.

ZFS doesn’t like single-drive pools, and also doesn’t like full drives, and the current blockchain size will be cutting it close. As the blockchain grows, I expect the pool’s performance to worsen.

I could buy another drive to set up a mirror, but storage is expensive and I don’t need two copies of the blockchain. If the SSD explodes tomorrow, I can follow this guide to recover my environment and sync the blockchain from scratch. When the size of the blockchain finally exceeds my 1TB SSD, the cheapest solution will be to extend the pool across a second 1TB drive. This is not an ideal setup, but all great tech spaghetti begins with yesterday’s budget consciousness, so at least I’ll have something to write an article about in the future!

Why Not Docker?

TrueNAS Scale has excellent Docker support, and I could use the community docker-bitcoind image https://apps.truenas.com/catalog/bitcoind/.

There are a few downsides to using Docker. First, iXSystems has pushed painful, breaking updates to the TrueNAS apps ecosystem in the past. This is not as troublesome for stateless microservices and simple web applications, but it could be a serious issue for a monolith like bitcoind.

Additionally, VM security and isolation is simply superior to Docker security and isolation. Bitcoin nodes are quite noisy and reach out to many third parties, by design, and containing this exposure within a VM is an appropriate isolation measure. Rather than configuring TrueNAS permissions for the dataset, running application, network stack, etc., we can just give bitcoind its own file system and users.

Furthermore, because we can use UEFI + virtio, our VM will be vendor-agnostic and can be easily ported to Proxmox or VMWare in the future.

Lastly, Docker apps on TrueNAS Scale route all disk I/O through ZFS, which is a huge performance hit for large sequential writes (like the initial block download). By using a VM on a ZVOL (raw block device), we can write these massive amounts of data without the copy-on-write ZFS overhead. This also avoids the file-level permissioning nightmare that commonly impacts TrueNAS Docker applications and are difficult to troubleshoot.

Creating the VM

First, create a ZFS pool so we have a place to store out VM and our data. Your hardware might be different, so differences at this stage can be safely ignored. Make sure you have enough storage for the current (and near-term future) size of the blockchain and the chainstate data. The result of my pool configuration is below:

Step two is to select an operating system for our purpose and download it. I chose Debian NetInst because it’s lightweight (low disk usage by the OS itself and low RAM usage at idle, ~400MB). You can also install NetInst without a GUI, which is unnecessary for this project and helps a lot on the resource side.

To download, visit Debian’s download page https://cdimage.debian.org/debian-cd/current/amd64/iso-cd/. Download the SHA256SUMS and debian-<version>-amd64-netinst.iso files. Take the extra security step to verify the integrity of the file. This isn’t foolproof, but it’s a good practice and something we’ll repeat more in-depth later. I downloaded version 13.3.0. Step three is to spin up the VM in the TrueNAS Scale web interface. In the sidebar, navigate to Virtualization. In the top-right corner, click the blue Add button. This will open a dialogue bar on the right-hand side of the screen which walks you through creating the VM. Change the Guest Operating System to Linux. Set the Name to whatever you want. Pick a good password. Everything else can stay as default. See my configuration below:

Next, we configure the resources we’re giving the VM. We need to distinguish between two phases of the Bitcoin node lifescycle: IBD (initial block download) and normal operation. During IBD, the node will consume as much CPU, RAM, I/O, and network as you give it, until it reaches a bottleneck somewhere. I most often see bottlenecks in CPU or disk write speed. During normal operation, your node will consume hardly anything, it’s very lightweight. The IBD can sometimes take north of a week if the node has a bottleneck somewhere. To achieve a faster IBD, you can increase the CPU resources, specifically Cores (the IBD does not benefit greatly from hyperthreaded cores aka Threads or virtual sockets aka vCPUs, it needs bare metal cores). If you choose to speed up the IBD by adding more Cores, make sure you reduce this back to one after the IBD is complete.

I gave my node one Core, Thread, and vCPU. I gave my node 4 GiB of RAM and set a minimum (reserved) RAM pool at 2 GiB (that guarantees at least 2 GiB will be available and TrueNAS will not choke the VM). See my configuration below:

Next, we’re going to assign the disk to bitcoin-pool (or whatever you named your pool) and set the size to 50 GiB. Make sure Create a new disk image is the selected radial option and Select Disk Type is set to VirtIO. See my configuration below:

Do not set the size to the whole disk size! Start with a small pool size to accomodate only the guest OS. Doing so will make the file system configuration much easier later.

Next, we need to tell TrueNAS how to give our VM network access. Set the Adapter Type to VirtIO, leave the MAC Address to the auto-generated one, and pick your host ethernet adapter interface from the Attach NIC dropdown list (you most likely will only have one option to choose). See my configuration below:

Next, we need to give TrueNAS the installation media for our operating system of choice. Click the Choose File upload button and navigate to the iso file you downloaded from Debian. This upload may take some time to complete, and the web interface may appear to be hung. Do not refresh or click back! You will have to repeat all the steps.

I highly encourage you to use the built-in upload button on the dialogue via the TrueNAS Scale web interface rather than trying to move the iso onto a network share. Even if you use the TrueNAS shell to move the file from the share onto an internal directory, you still have to reconfigure the permissions for all different processes and utilities that need to touch it during the VM creation process. TrueNAS will do all of this for you if you just upload the iso directly.

Configuring Boot and Disks

After the initial set up, we need to remove the iso to get our VM to boot into the actual OS and not the installation media.

Navigate to the TrueNAS Virtualization page and click the Devices button on your VM. You will see four entries. Click the three dots on the right-hand side corresponding to the CD-ROM entry (which should also have Order of 1000) and click Delete:

Click Delete Device:

Now, reboot the VM.

After the VM has rebooted, click the Display button. You should now see the Debian login screen instead of the set up menus:

We now have a working Debian VM! Shut it down so we can make a few changes.

Next, we need to give our VM access to the remainder of the storage space from our pool. Navigate to TrueNAS > Datasets and find the pool we created earlier. The pool should have one entry: a ZVOL with the name of your VM as the prefix. Click on the ZVOL then click Edit Zvol.

Change the Size for this Zvol to the maximum that you can. For me that was 880 GiB. Check the Force size box and click Save:

Power the VM back on. Now it’s time to SSH into the VM. Depending on how your network is configured, you may be able to do this using the hostname you configured in the initial set up. Now, to create the actual mount point, we need to get the internal ID of the vda4 partition. Technically you can use the device name of /dev/vda4, but these assignments are at the mercy of the OS and might change. For a truly stable mount, we need to use the UUID. To get it, run sudo blkid /dev/vda4:

1
2

taylor@debian:~$ sudo blkid /dev/vda4
/dev/vda4: UUID="f7461174-1a26-4590-a1d0-6f2b2d63189f" BLOCK_SIZE="4096" TYPE="ext4" PARTUUID="2e96b7ee-c584-463a-9fe1-b2a845b93811"

Now that we have the UUID f7461174-1a26-4590-a1d0-6f2b2d63189f, we need to edit the /etc/fstab file. We will use a text editor called Nano to make these this small change. The fstab file (filesystem table) tells linux when, where, and how to create mount points to the underlying file system. In order to access our newly formatted partition via a path name like /mnt/bitcoin, we need to make sure linux will reliably create the link between the two.

1

taylor@debian:~$ sudo nano /etc/fstab

Once you’re inside Nano editing the /etc/fstab file, scroll to the bottom and add a new entry. I recommend leaving a comment above by using the pound symbol # and saying when and why you created this mount point. See the template below, you can copy it excplicitly and change the UUID:

1
2

# new mount point for /mnt/bitcoin -taylor 2026-03-01
UUID=f7461174-1a26-4590-a1d0-6f2b2d63189f /mnt/bitcoin ext4 defaults,noatime,nofail 0 2

When you’re finished, hit Ctrl + X then Y to exit out of Nano while saving your changes.

Let’s check success. Use sudo mount -a to force Linux to recreate all the mount points, then df -h /mnt/bitcoin to see the status of the new mount we just created:

Installing and Configuring Bitcoin Core

Before we begin, what does it mean to “Download Bitcoin”? There are competing forks of Bitcoin and competing versions (0.1 through 30 at the time of writing), so which one should you pick? If you want your node to support the critical soft forks that are active today, SegWit and Taproot, then you need at least v22.0. (SegWit support was introduced much earlier, but Taproot enforcement was first fully implemented in v22.0.). However, Bitcoin version updates aren’t for fun, each one brings important security, reliability, or performance upgrades. So, the latest version? Well, many Bitcoiners are not happy with the way changes were made for the most recent release, Bitcoin Core v30. A controversial change was made despite significant pushback (you can read about OP_RETURN and make up your own mind), which spawned the competing fork known as Bitcoin Knots.

The answer to this question has to come from your reason to do this in the first place. I need a node to work on software that will be talking to a node, and if I want that software to be useful to others, I need to make sure it’s compatible with newer versions. For this reason, I am downloading v28, which was released in October 2024. The instructions that follow should be largely version-agnostic.

Before we unpack the archive files, we need to verify that they haven’t been tampered with.

The basic idea here is that we can hash the archive we downloaded and compare it with the hash that the developers have provided. This is an excellent checking mechanism, but if an attacker managed to tamper with the archive file, they could easily just re-compute the hash then tamper with the hash file as well. So how do we make sure the hash file hasn’t been tampered with? The developers also provide a digitally signed version of the SHA256SUMS file. Anyone can hash, but only the developers can sign.

Finally, check the bitcoin core download itself.

1
2

taylor@debian:~$ sha256sum --check SHA256SUMS --ignore-missing
bitcoin-28.0-x86_64-linux-gnu.tar.gz: OK

To recap, we downloaded Bitcoin Core v28.0, a list of hashes for all v28.0 files, a signature file which proves the hashes are authentic, and the keys for all Bitcoin Core signers which proves the signature file is authentic. In order for an attacker to tamper with this download, they would have to compromise the public GitHub listing of all the developers keys, replace all the keys with fake keys, create a modified version of the release, hash it, sign the hashes with the fake keys, then somehow get the fake versions onto the Bitcoin Core file server. Doing any one of those things would raise an army of red flags, so I think we’re safe.
